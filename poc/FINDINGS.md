# vime.vim PoC 問題洗い出し結果

英字→日本語変換 Vim プラグインの feasibility 検証で、考えられるパターンを PoC して洗い出した問題の一覧。

検証環境: macOS 26 (Apple Silicon) / Neovim 0.12.2 (LuaJIT 2.1) / anthy-9100h (nix) / libanthy.dylib を LuaJIT FFI で直叩き。

PoC スクリプト:
- `poc/ffi_poc.lua` — FFI 疎通
- `poc/ffi_quality_poc.lua` — 変換精度 44 パターン
- `poc/ffi_chunk_poc.lua` — 一括 vs ユーザー分割変換
- `poc/romaji.lua` / `poc/romaji_poc.lua` — ローマ字→かな + end-to-end
- `poc/problem_sweep.lua` — エッジケース網羅スイープ
- `poc/anthy_advanced_poc.lua` / `poc/anthy_learning_poc.lua` — 文節伸縮・学習
- `poc/ime_render_poc.lua` — インライン未確定描画(下線/文節反転/popup/キー横取り)
- `poc/latency_poc.lua` — 変換レイテンシ測定

## 洗い出した問題

### A. ローマ字→かな変換

| ID | 問題 | 重要度 | 状態/方針 |
|----|------|--------|-----------|
| A1 | `ん` の扱い: `nn` を常に2文字消費すると `こんにちは→こんいちは`, `おんな→おんあ` になる | 高 | **修正済**。2つ目の n の次が母音/y なら n を1つだけ消費する look-ahead 規則に変更 |
| A2 | 大文字未処理: `Kyou→Kよう`, `KANJI→KANJI` | 高(要判断) | 設計判断。(a)全部小文字化 (b)SKK風に大文字を変換境界の合図に使う、の二択 |
| A3 | 外来音未対応: `va→vあ`, `tsa→tさ`, `di→ぢ`(modernはでぃ), `ti→ち` | 中 | テーブル拡張で対応可。`di/ti` は kunrei/modern で割れるので方針決めが必要 |
| A4 | Hepburn 式 `m` 撥音: `namba→なmば`(難波は nanba 必須) | 低 | 実IME準拠。`honya→ほにゃ` も同様に正しい挙動(本屋は hon'ya) |
| A5 | 記号・句読点: `a,b→あ,b` のまま | 低(要判断) | `、。` へ変換するか英数記号 passthrough か |

### B. Anthy 変換（入力の前処理が必要）

| ID | 問題 | 重要度 | 方針 |
|----|------|--------|------|
| B1 | 空文字列で `nr_segment=0` | 中 | `get_segment` を呼ぶ前に 0 判定のガードが必要 |
| B2 | 英数字/記号がそのまま素通り(`abc→abc`, `123→123`) | 中 | 変換対象を「かなのみ」に限定する前処理が必要 |
| B3 | 全角記号を anthy が勝手に半角化(`！？→!?`) | 低 | 記号正規化の挙動として把握しておく |

### C. FFI / 堅牢性（おおむね良好）

| ID | 内容 | 方針 |
|----|------|------|
| C1 | 文節は常に短い(1バンセツ)。超長文1000かなでも bufsize 256 で切り詰め無し | 安全だが正攻法は C2 で必要長を取得して動的確保 |
| C2 | `anthy_get_segment(...,nil,0)` で必要バイト長が取れる | 動的バッファ確保の作法が確定 |
| C3 | 範囲外 `seg=999,cand=999` でもクラッシュしない | anthy が防御的。呼び出し側でも境界チェック推奨 |
| C4 | `set_string` で前回状態は正しくリセットされる | context 再利用は安全 |

### D. 統合 UX / 運用（設計フェーズで詰める）

| ID | 課題 | メモ |
|----|------|------|
| D1 | `.so` パスのハードコード(nix store path) | 可搬性。設定注入 or 探索が必要。公開時は利用者の anthy 導入前提 |
| D2 | 学習を効かせるには**全文節 commit が必須** | 部分 commit では学習されない。確定 UX を「文単位」に寄せる必要 |
| D3 | 学習は `~/.anthy` に永続化される(副作用) | PoC でも実際に書き込み済み。利用者の学習 DB を育てる前提でよい |
| D4 | 変換対象テキストの範囲特定 | 挿入モードで「どこからどこまでがローマ字未確定文字列か」を管理する仕組みが要る |
| D5 | 未確定状態の UI 表現 | **検証済**: extmark で下線(未確定)・反転(文節)・floating popup(候補)・挿入モードキー横取り すべて可。**罠**: 文節ハイライトは byte offset 必須(日本語1文字=3byte、文字数を使うとズレる) |

---

## 設計に直結する結論

1. **方式は IME 的なユーザー分割変換を主軸にする**のが最適。区切りミスが消え候補選択到達 100%。長く打てば文脈活用も可能。
2. **resize_segment / commit_segment は両方使える**。一括変換時の区切りミス回収と、誤変換の学習矯正が可能。
3. **学習は文単位確定で記録**される。確定 UX の単位設計に直結。
4. 残る設計判断: 大文字の扱い(A2)、変換対象範囲の管理(D4)、未確定 UI(D5)。
